## Pipeline flow (as built)

```
User input
  → Ollama llama3.2: extract product data (shipping now null-safe)
  → Groq llama-3.3-70b-versatile: classify HTS code
     └─ fallback: hts-category-map.json (22 categories)
  → USITC exportList API: fetch MFN base rate
     └─ USMCA fast-path: skip API, return 0
  → apply COUNTRY_SURCHARGES (China +25%, Vietnam +20%)
  → Ollama llama3.2: analysis with margin formulas + retailer context
```

2. **USITC exportList returns 0 results for 8-digit codes** — Groq sometimes returns 8-digit codes like `6109.10` but the USITC exportList needs 10-digit codes for exact matches. When this happens, `lookupTariffRate` returns null and the model estimates. Could try appending `.00` or `.00.00` to get a match, or use the `from/to` range trick (`from=6109.10&to=6109.99`). ## Decisions made

| Decision | Rationale |
|----------|-----------|
| Groq for HTS classification, not Ollama | llama3.2 3B is too small for accurate 10-digit HTS code lookup; llama-3.3-70b-versatile on Groq gets it right reliably |
| Option B: keep 5 broad extraction categories | llama3.2 3B would hallucinate from a 18-item category list, causing fallback misses at demo time |
| `shipping_cost_per_unit: number \| null` | Silent `?? 0` default was hiding extraction failures and showing $0 shipping in results |
| Derive wholesale from margin formula | `wholesale_price = landed_cost / (1 - supplier_margin / 100)` prevents the model from setting wholesale = landed_cost (0% margin) |
| `HTS_CATEGORY_MAP` no longer exported | Nothing outside `hts-client.ts` imported it |
| `npx tsx` over `ts-node` | `ts-node` doesn't work with `"type": "module"` ESM projects |
